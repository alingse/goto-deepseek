### 1.2 为什么单线程能高性能？

```
┌─────────────────────────────────────────────────────────────┐
│            单线程高性能的原因分析                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   1. 内存操作，O(1) 复杂度                                │
│      - dictFind, sdsLen, listPop 都是 O(1)               │
│      - 纯内存操作，无磁盘 I/O 阻塞                         │
│                                                             │
│   2. C 语言，高效执行                                      │
│      - 无 GC 暂停 (引用计数)                               │
│      - 直接内存操作                                        │
│                                                             │
│   3. I/O 多路复用                                         │
│      - epoll/kqueue/select                                │
│      - 单线程处理海量连接                                  │
│                                                             │
│   4. 无锁竞争                                              │
│      - 无需上下文切换                                      │
│      - 无锁争用开销                                        │
│                                                             │
│   实际瓶颈:                                                │
│   - 网络读取/写入 (非阻塞但有开销)                         │
│   - 协议解析                                               │
│   - 大 Key 操作                                            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 二、Redis 6.0 I/O Threads

### 2.1 架构设计

```
┌─────────────────────────────────────────────────────────────┐
│                   Redis 6.0 I/O Threads                    │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   主线程 (Main Thread):                                    │
│   ┌─────────────────────────────────────────────────────┐ │
│   │ 1. 监听端口 (epoll)                                  │ │
│   │ 2. 接受连接                                          │ │
│   │ 3. 分配 client 给 I/O 线程                         │ │
│   │ 4. 解析命令                                          │ │
│   │ 5. 执行命令                                          │ │
│   │ 6. 将响应放入缓冲区                                  │ │
│   └─────────────────────────────────────────────────────┘ │
│                           │                                 │
│                           ▼                                 │
│   ┌──────────┐  ┌──────────┐  ┌──────────┐             │
│   │  I/O     │  │  I/O     │  │  I/O     │             │
│   │ Thread 1 │  │ Thread 2 │  │ Thread 3 │             │
│   │          │  │          │  │          │             │
│   │ - 读取   │  │ - 读取   │  │ - 读取   │             │
│   │ - 写入   │  │ - 写入   │  │ - 写入   │             │
│   └──────────┘  └──────────┘  └──────────┘             │
│                                                             │
│   配置:                                                    │
│   io-threads 4        # 4 个 I/O 线程                    │
│   io-threads-do-reads yes  # 启用多线程读取              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 工作流程

```c
// src/server.c - I/O 线程处理流程

/*
 * I/O 线程工作流程:
 *
 * 1. 主线程 accept 新连接
 * 2. 将 client 加入等待队列
 * 3. I/O 线程从队列取任务
 * 4. I/O 线程执行:
 *    - readQueryFromClient() (如果启用)
 *    - writeToClient() (写回响应)
 * 5. 主线程执行命令
 */
```

### 2.3 配置与启用

```bash
# redis.conf

# 启用 I/O 线程
io-threads 4

# 让 I/O 线程也参与读取 (默认只写)
io-threads-do-reads yes

# 注意事项:
# - 建议设置 CPU 核心数 - 1
# - 过多线程会增加复杂度
# - 命令执行仍是单线程
```

## 三、Redis 7.2+ 多线程增强

### 3.1 Client Output Buffer 多线程

```
┌─────────────────────────────────────────────────────────────┐
│         Redis 7.2 Client Output Buffer 线程化              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   传统模型:                                                │
│   ┌─────────────┐                                        │
│   │   主线程    │ ──► 写入响应到客户端                   │
│   └─────────────┘                                        │
│        │                                                  │
│        ▼                                                  │
│   主线程阻塞等待 socket 写入完成                           │
│                                                             │
│   Redis 7.2+ 模型:                                        │
│   ┌─────────────┐      ┌─────────────────┐               │
│   │   主线程    │ ──►  │ Output Buffer  │               │
│   └─────────────┘      │   Thread       │               │
│                        └─────────────────┘               │
│                              │                             │
│                              ▼                             │
│                        异步写入到客户端                     │
│                                                             │
│   收益:                                                    │
│   - 主线程不再阻塞于网络写入                               │
│   - 高吞吐场景下提升明显                                  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 后台任务线程化

```
┌─────────────────────────────────────────────────────────────┐
│              Redis 7.2 后台任务线程化                      │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   后台任务:                                                │
│   ┌─────────────────────────────────────────────────────┐ │
│   │ 1. AOF 写入 (everysec 模式)                        │ │
│   │    - 原来: 主线程同步写入                            │ │
│   │    - 现在: 后台线程异步写入                          │ │
│   │                                                     │ │
│   │ 2. 惰性删除 (lazy free)                           │ │
│   │    - 原来: 主线程同步释放大对象                      │ │
│   │    - 现在: 后台线程释放                             │ │
│   │                                                     │ │
│   │ 3. 统计信息收集                                     │ │
│   │    - 后台线程定时更新统计                           │ │
│   └─────────────────────────────────────────────────────┘ │
│                                                             │
│   效果:                                                    │
│   - 主线程更专注于命令处理                                │
│   - 减少延迟毛刺                                          │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3.3 无锁数据结构改进

```c

 *    - 改进的消息队列
 *    - 减少锁竞争
```

## 四、Redis 8.x 架构优化

### 4.1 RFCH (Redesigned Fast Cursor Hash)

```
┌─────────────────────────────────────────────────────────────┐
│               Redis 8.0 RFCH 改进                         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   问题:                                                    │
│   - 传统 hash 表 rehash 时需要遍历旧表                    │
│   - 大 hash 表 rehash 会有延迟毛刺                        │
│                                                             │
│   RFCH 改进:                                              │
│   ┌─────────────────────────────────────────────────────┐ │
│   │ 1. 新的渐进式 hash 实现                             │ │
│   │    - 不需要全量遍历                                 │ │
│   │    - 每个操作只迁移必要的数据                       │ │
│   │                                                     │ │
│   │ 2. 内存效率提升                                     │ │
│   │    - 更紧凑的存储格式                               │ │
│   │    - 减少指针开销                                  │ │
│   │                                                     │ │
│   │ 3. 性能提升                                        │ │
│   │    - hash 操作延迟降低 20-30%                      │ │
│   │    - 内存使用降低 10-15%                           │ │
│   └─────────────────────────────────────────────────────┘ │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 TSAN (Thread-Local Auxiliary NUMA)

```c
/*
 * Redis 8.0 NUMA 感知优化
 *
 * 问题:
 * - 多核服务器上，跨 NUMA 访问内存延迟高
 *
 * 解决方案:
 * - 为每个 NUMA 节点分配本地内存池
 * - 线程优先使用本地节点内存
 *
 * 配置:
 * numactl --cpunodebind=0 redis-server
 * 或
 * redis-server --numa true
 */
```

## 五、多线程配置最佳实践

### 5.1 配置建议

```bash
# redis.conf

# 通用建议
io-threads 4              # CPU 核心数 - 1
io-threads-do-reads yes  # 启用多线程读取

# 如果有大量写入
io-threads 6             # 写入多时增加

# AOF 配置
appendonly yes
appendfsync everysec     # 每秒同步

# 内存
maxmemory 4gb           # 根据实际设置
maxmemory-policy allkeys-lru
```

### 5.2 性能测试

```bash
# 使用 redis-benchmark 测试
redis-benchmark -h localhost -p 6379 -c 50 -n 100000 -P 16

# 参数说明:
# -c: 并发连接数
# -n: 总请求数
# -P: 每个连接 Pipeline 请求数
# -t: 只测试指定命令

# 测试不同配置
redis-benchmark -t set,get -c 100 -n 100000 --threads 4
```

## 八、常见问题

### Q1: Redis 6.0 多线程是否线程安全？

```
是的，线程安全:
- I/O 线程只负责读写
- 命令执行仍在主线程
- 使用原子操作和锁保护共享数据
```

### Q2: 开启多线程后性能提升不明显？

```
可能原因:
1. 命令执行是瓶颈 (单线程)
2. 网络带宽饱和
3. 客户端数量少
4. 操作本身很快，多线程开销反而更大

建议:
- 使用 Pipeline 批量操作
- 评估实际瓶颈在哪里
```

### Q3: 多线程模式是否稳定？

```
Redis 6.0+ 多线程经过多年生产验证:
- 稳定性与单线程版本相当
- 官方推荐生产使用
- 但默认仍关闭，需要手动启用
```

### Q4: Redis 7 的多线程 vs KeyDB/Dragonfly？

```
KeyDB/Dragonfly:
- 更激进的多线程策略
- 全命令多线程执行
- 但不是 Redis 官方版本

Redis:
- 保守渐进式改进
- 保持核心简单性
- 社区认可度高
```
