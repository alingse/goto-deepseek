## 一、Redis 核心数据结构深度解析

### 1.1 String（字符串）

#### 底层实现：SDS (Simple Dynamic String)

```c
// Redis 源码中的 SDS 结构
struct sdshdr {
    int len;        // 已使用长度
    int free;       // 剩余可用长度
    char buf[];     // 柔性数组，存储实际数据
};
```

#### 为什么 String 是 O(1) 复杂度？

```
┌─────────────────────────────────────────────────────────────┐
│                     Redis String 访问                        │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   GET/SET 操作流程：                                        │
│   1. 根据 key 计算 hash 值                                  │
│   2. 找到对应的 dictEntry (O(1) hash 查找)                 │
│   3. 获取 sdshdr 指针，直接操作 buf 数组                    │
│   4. 读取 len 直接返回，无需遍历                             │
│                                                             │
│   关键点：                                                  │
│   - Redis 内部维护全局 hash 表 (dict)                       │
│   - 每个 key-value 对应一个 dictEntry                       │
│   - value 的指针指向 sdshdr 结构                            │
│   - 读取/修改 len 字段即可知道字符串长度                    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### String 的特性与适用场景

| 特性 | 说明 |
|-----|-----|
| **二进制安全** | 可以存储任意二进制数据（图片、序列化对象） |
| **动态扩容** | 预分配空间，减少频繁内存分配（空间换时间） |
| **长度缓存** | len字段缓存长度，避免strlen O(n) 问题 |
| **惰性释放** | 释放空间时不立即回收，保留供下次使用 |

**典型场景**：
- 缓存 JSON 序列化对象
- 分布式锁（SET NX EX）
- 计数器（INCR/INCRBY）
- Session 存储

### 1.2 Hash（哈希表）

#### 底层实现：ziplist vs hashtable

```
数据量小 (<512项)         数据量大 (>=512项)
     │                          │
     ▼                          ▼
┌─────────┐              ┌─────────────┐
│ ziplist │              │  hashtable  │
│ 压缩列表│              │   哈希表     │
└─────────┘              └─────────────┘
     │                          │
     ▼                          ▼
  O(n) 遍历              O(1) 操作
  内存紧凑               查找快
```

#### 为什么 Hash 能做到 O(1) 查询？

```c
// Redis 源码中的 dict 结构
struct dict {
    dictType *type;
    dictEntry **table;      // 哈希表数组
    unsigned long size;     // 哈希表大小
    unsigned long used;     // 已使用条目数
    ...
};

// 哈希表查找过程
dictEntry *dictFind(dict *d, const void *key) {
    // 1. 计算 hash 值: hash(key)
    // 2. 定位桶: index = hash & (size-1)  (与运算替代取模)
    // 3. 遍历桶链表 (或跳表) 找到 exact match
    // 4. 平均 O(1)，最坏 O(n) (哈希碰撞)
}
```

```
┌─────────────────────────────────────────────────────────────┐
│                    Hash 查找 O(1) 原理                       │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   key = "user:1"                                           │
│                                                             │
│   Step 1: hash("user:1") = 0x7f3a...  (64位hash)          │
│                                                             │
│   Step 2: index = hash & (size-1)                         │
│            假设 size=4 (桶数量)                             │
│            index = 0x7f3a... & 0x03 = 2 (桶编号)           │
│                                                             │
│   Step 3: 直接访问 table[2]，检查 key 是否匹配             │
│                                                             │
│   为什么快？                                                │
│   - 与运算比取模快无数倍                                    │
│   - 数组随机访问 O(1)                                      │
│   - 哈希函数分布均匀，单个桶平均元素少                       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### Hash 的 rehash 机制（渐进式）

```c
// Redis 使用两个 hash 表，实现渐进式 rehash
struct dict {
    dictEntry **ht_table[2];   // 两个哈希表
    unsigned long ht_used[2];  // 每个表的已用数量
    long rehashidx;           // rehash 进度，-1表示未开始
};

// 渐进式 rehash 过程：
// 1. 新请求来时，迁移 1 个 bucket
// 2. 每次迁移完成后 rehashidx++
// 3. 期间读写操作同时访问两个表
// 4. 迁移完成后切换新旧表
```

#### Hash 适用场景

| 场景 | 优势 |
|-----|-----|
| 对象存储 | 避免序列化整个对象，按字段更新 |
| 购物车 | HINCRBY 更新商品数量 |
| 统计计数 | 多个维度计数的组合 |

### 1.5 Sorted Set（有序集合）⭐ 重点

#### 底层实现：ziplist vs skiplist + dict

```
元素少 (<128个, 每个<64字节)      元素多 (>=128个)
         │                                 │
         ▼                                 ▼
   ┌─────────────┐                 ┌─────────────────┐
   │   ziplist   │                 │  skiplist + dict│
   │  (压缩列表)  │                 │   (跳表+字典)   │
   └─────────────┘                 └─────────────────┘
   两元素配对: [score, member]         跳表: 按score排序
   按score升序排列                     字典: 按member查O(1)
```

#### 为什么 ZSET 能同时做到 O(1) 查 member 和 O(log n) 查排名？

```c
// Redis 源码中的 zset 结构
typedef struct zset {
    dict *dict;           // 字典: member -> score (O(1) 查找)
    zskiplist *zsl;      // 跳表: 按 score 排序 (O(log n) 范围查询)
} zset;

// 查询 member 的 score：dictFind(dict, member) -> O(1)
// 查询某排名范围的元素：zslRangeByRank() -> O(log n + result_count)
```

```
┌─────────────────────────────────────────────────────────────┐
│              ZSET 双向索引结构 (skiplist + dict)            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   跳表 (按 score 排序)                                      │
│   ┌─────────────────────────────────────────────┐          │
│   │  L3:  [score:10] ──────────────────────────► NULL    │
│   │  L2:  [score:10] ──────► [score:50] ──────► NULL    │
│   │  L1:  [score:10] ─►[30]►[score:50] ─►[80]► NULL    │
│   └─────────────────────────────────────────────┘          │
│                                                             │
│   字典 (按 member 查找)                                     │
│   ┌─────────────────────────────────────────────┐          │
│   │  dict: {"alice" -> 10, "bob" -> 50, ...}  │          │
│   └─────────────────────────────────────────────┘          │
│                                                             │
│   查询 alice 的分数: dictFind(dict, "alice") = 10 (O(1))   │
│   查询前10名: zslRangeByRank(zsl, 1, 10) = O(log n)        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### ZSET 适用场景

| 场景 | 使用方式 |
|-----|---------|
| 排行榜 | ZADD + ZREVRANGE |
| 延迟队列 | ZADD + ZRANGEBYSCORE |
| 权重任务队列 | ZADD + ZRANGE |

### 2.1 槽（Slot）分配机制

#### 16384 个槽的由来

```
┌─────────────────────────────────────────────────────────────┐
│              为什么是 16384 个槽？                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   1. CRC16(key) mod 16384                                  │
│                                                             │
│   2. 槽数量选择考虑因素:                                    │
│      - 节点数: Redis Cluster 最大 16384 节点               │
│      - 槽多=迁移快: 每个槽独立迁移                          │
│      - 槽少=网络包小: 节点信息交换 (Gossip)                 │
│                                                             │
│   3. 实际限制:                                              │
│      - 建议最多 1000 节点 (超过网络开销大)                  │
│      - 一般 3 主 3 从，共 6 节点                            │
│      - 16384 远大于实际节点数，足够细粒度                    │
│                                                             │
│   4. 槽迁移时:                                              │
│      - MOVED 错误告诉客户端正确节点                          │
│      - ASK 错误告诉客户端去目标节点尝试                      │
│      - 客户端本地缓存槽->节点映射                           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 三、RedLock 分布式锁

### 3.1 为什么需要 RedLock？

```
┌─────────────────────────────────────────────────────────────┐
│              单节点 Redis 锁的问题                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   SET lock:123 NX EX 30                                    │
│                                                             │
│   问题场景:                                                 │
│   ┌─────────┐    网络分区    ┌─────────┐                   │
│   │ Client A│ ──────────────►│  Redis  │                   │
│   └─────────┘                └─────────┘                   │
│        │                            │                       │
│        │ 获得锁成功                 │                       │
│        ▼                            ▼                       │
│   ┌─────────┐                       │                       │
│   │执行业务 │                       │ Redis 主节点宕机      │
│   │ 30秒    │                       ▼                       │
│   └─────────┘                  ┌─────────┐                 │
│        │                       │ (主从)  │                 │
│        ▼                       │ 从节点  │                 │
│   释放锁 (DEL) ──────────►    │ 无锁    │                 │
│                               └─────────┘                 │
│   结果: Client B 获得锁，但 Client A 还在执行业务            │
│                                                             │
│   结论: 主从切换导致锁丢失                                  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 RedLock 算法原理

```
┌─────────────────────────────────────────────────────────────┐
│                   RedLock 算法                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   假设: N=5 个 Redis 节点 (独立)                            │
│                                                             │
│   获取锁流程:                                               │
│   ┌─────────────────────────────────────────────────────┐  │
│   │ 1. 获取当前时间 T1                                   │  │
│   │ 2. 依次向 5 个节点执行 SET lock NX EX 30            │  │
│   │ 3. 记录成功获取锁的节点数 M                           │  │
│   │ 4. 如果 M >= 3，且 (T2-T1) < lock_expire_time        │  │
│   │    则成功获取锁                                       │  │
│   │ 5. 否则获取失败，向所有节点发送 DEL                   │  │
│   └─────────────────────────────────────────────────────┘  │
│                                                             │
│   为什么需要多数节点同意:                                    │
│   - 5 个节点中最多 2 个宕机，仍能正常工作                   │
│   - 避免单点故障                                            │
│                                                             │
│   为什么需要时间检查:                                        │
│   - 防止部分节点响应慢导致锁超时                            │
│   - 确保锁有效期内完成                                      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3.3 RedLock 的问题与争议

```
┌─────────────────────────────────────────────────────────────┐
```

### 3.4 最佳实践

```bash
# 使用 Redisson 实现分布式锁
RLock lock = redissonClient.getLock("myLock");
lock.lock();
try {
    // 业务逻辑
} finally {
    lock.unlock();
}

# 特性:
# - 看门狗自动续期 (默认 30s，检查周期 1/3)
# - 可重入
# - 公平锁 (FIFO)
# - 联锁 (MultiLock)
```

## 五、常见面试题汇总

### Q1: Redis 为什么快？

1. 内存存储，O(1) 操作
2. 单线程 + I/O 多路复用
3. C 语言实现，执行效率高
4. 减少上下文切换和锁竞争

### Q2: Redis 数据结构选型？

| 场景 | 推荐数据结构 |
|-----|-------------|
| 缓存对象 | Hash |
| 排行榜 | Sorted Set |
| 消息队列 | Stream / List |
| 去重 | Set / Bloom Filter |
| 计数器 | String |
| 限流 | String + Lua |

### Q3: Redis 持久化选型？

| 持久化方式 | 适用场景 |
|-----------|---------|
| RDB | 冷备份、快速恢复 |
| AOF | 数据完整要求高 |
| 混合 | 最佳平衡 (4.0+) |

### Q4: Redis 淘汰策略选型？

| 策略 | 适用场景 |
|-----|---------|
| volatile-lru | 有明确过期时间的数据 |
| allkeys-lru | 所有数据平等淘汰 |
| noeviction | 不允许淘汰（写入型） |
