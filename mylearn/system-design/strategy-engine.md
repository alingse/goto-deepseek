# 高性能过滤引擎与规则沙箱深度指南

本文档深入探讨史斌 CV 中“机审引擎重构”部分。作为资深 Go 后端，如何在百万级关键词和数千条动态规则下，将 QPS 从 5k 提升至 20k，并将延迟控制在 10ms 以内。

## 1. 核心技术深度拆解

### 1.1 AC 自动机的内存与性能极限优化
*   **挑战**：传统的 AC 自动机在关键词达到百万级时，Trie 树节点爆炸，内存占用高且 CPU Cache Miss 严重。
*   **细节**：
    *   **Double Array Trie (DAT)**：将指针型 Trie 树压缩为两个线性数组，大幅提升空间利用率和 CPU 缓存命中率。
    *   **失败指针压缩**：通过路径压缩减少失败路径的跳转次数。
    *   **冷热 Hash 分层**：将超高频关键词留在 Aho-Corasick 树中，低频词采用分块匹配。

### 1.2 类型安全与程序沙箱隔离
*   **细节**：如何防止业务人员写的“坏规则”拖垮整个 Go 进程？
    *   **DSL 编译**：不直接执行代码，而是将规则编译为特定的字节码或抽象语法树（AST）。
    *   **运行时限制**：在 Go 内部实现一个极简的 VM，限制每个规则的指令执行数（Instruction Limit）和内存分配量。

## 2. 专家级面试对垒

### 场景一：AC 自动机在大规模动态更新下的“锁”问题
**面试官 (Expert)**：
> “你的系统支持 20k QPS。当运营人员需要实时更新数万个关键词时，如果直接锁住整棵 AC 树进行重建，会导致严重的延迟毛刺。你如何在不停机、不丢包的前提下完成秒级更新？”

**候选人 (Senior Engineer)**：
> “这是一个典型的**‘读多写少’且‘写极其沉重’**的场景。我采用 **Copy-on-Write (CoW) + 双缓冲（Double Buffering）策略**：
> 1. **双缓冲结构**：内存中始终维护两棵 AC 树（Active 和 Shadow）。
> 2. **异步构建**：新关键词到达后，由后台协程在 Shadow 树上进行增量构建或全量重建，这个过程完全不影响 Active 树的读取。
> 3. **原子切换**：构建完成后，通过 `atomic.StorePointer` 瞬间切换指针。
> 4. **GC 友好**：为了防止大规模切换导致的 GC 压力，我会使用 **内存池（sync.Pool）** 或预分配的大切片来复用 DAT 数组，避免在堆上频繁分配百万级的小对象。”

### 场景二：规则执行的优先级抢占与资源隔离
**面试官 (Expert)**：
> “在内容安全场景下，‘色情’、‘暴恐’规则必须毫秒级响应，而‘广告’或‘垃圾话’规则可以慢一点。当系统过载时，你如何确保高危规则优先执行并抢占 CPU 资源？”

**候选人 (Senior Engineer)**：
> “在 Go 的运行时（Runtime）层面，协程是协作式调度的，很难做严格的物理抢占。我是在**业务执行器（Executor）层面**实现的逻辑抢占：
> 1. **规则分层**：将规则标记为 P0 (致命), P1 (普通), P2 (低优先级)。
> 2. **并行执行器**：
>    *   P0 规则在专门的‘快车道’协程池执行，预留固定配额的 CPU 核心。
>    *   P1/P2 规则使用普通的 Worker 池。
> 3. **中断机制**：一旦 P0 规则命中并判定为‘拦截’，执行器立即通过 `context.Done()` 发送信号给该请求下正在运行的所有 P1/P2 规则，强制中断后续不必要的计算。
> 4. **时间配额（Time Quota）**：每个请求的总处理时间是固定的（如 50ms）。我会先跑 P0 扫描，如果 P0 耗时超出阈值，直接截断后续规则并触发兜底保护。”
