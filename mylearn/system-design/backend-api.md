# 高并发服务端与 AI Chat Bot 架构面试指南

本文档结合 `learn.md` 的系统设计要点、JD 中的 AI Chat Bot 要求以及史斌（alingse）的资深 Go 后端经验，从高级专家面试官与高级资深工程师的角度进行深度对垒。

## 1. 核心摘要与深度细节补充

### 1.1 API 网关与 WebSocket 长连接
*   **细节补充**：在 AI Chat 场景下，传统的 HTTP 短链无法满足流式输出（Streaming）的需求。网关层需支持 **SSE (Server-Sent Events)** 或 **WebSocket**。
*   **网关挑战**：长连接状态保存在哪？网关重启如何平滑迁移连接？如何做基于 UserID 的路由？

### 1.2 缓存策略：从传统缓存到语义缓存
*   **细节补充**：除了 Redis 的大 Key、热 Key 处理，AI 场景需要 **Semantic Cache**（语义缓存）。
*   **穿透与限流**：针对昂贵的 LLM 调用，限流需在网关层（令牌桶）和业务层（基于 Token 消耗）双重实现。

### 1.3 分布式 Single-flight
*   **细节补充**：当多个用户提问同一个热点问题时，如何确保只向大模型发起一次请求？本地 `singleflight` 无法解决集群问题，需结合 Redis 分布式锁实现。

## 2. 面试对垒 (Expert vs. Senior)

### 场景一：AI Chat Bot 的高并发与高延迟矛盾
**面试官 (Expert)**：
> “大模型的推理延迟通常在秒级，而我们的产品有数千万 DAU。在高并发场景下，如果大量请求堆积在后端，会导致线程池耗尽或连接数溢出。作为资深架构师，你如何设计一套架构来兼顾‘高吞吐’和‘流式低延迟体验’？”

**候选人 (Senior Engineer)**：
> “首先，核心思路是**全链路异步化与流式响应**。
> 1. **连接管理**：在网关层采用 WebSocket 或 SSE，将请求解耦。用户发起提问后，后端立即返回 RequestID，进入 Pulsar 异步队列。
> 2. **异步调度**：利用我之前在知乎做的‘分层漏斗架构’。Pulsar 负责削峰填谷，Worker 节点根据大模型算力配额拉取任务。
> 3. **流式回传**：Worker 调用 LLM 得到一部分 Token 后，立即通过 Redis Pub/Sub 或专用网关的回调接口，将数据推送到对应的长连接。
> 4. **资源保护**：针对 LLM 这种慢速后端，必须配置**严格的限流和熔断**。我会设计一套基于‘预计消耗 Token’的动态限流算法，而不是简单的 QPS 限流，防止模型过载导致的雪崩。”

**追问 (Expert)**：
> “如果用户在等待流式输出时断开了连接，你会如何处理正在进行的模型推理任务？如何避免算力浪费？”

**回答 (Senior Engineer)**：
> “这是典型的‘无效计算’问题。
> 1. **Context 传递**：在 Go 内部，我们会全程传递 `context.Context`。当网关感知到 WebSocket 断开时，通过 RPC 调用或信号机制通知下游 Worker 触发 `cancel`。
> 2. **状态检查**：Worker 在处理循环（如迭代生成 Token 的 Loop）中，每输出一个 Chunk 前，都会检查 Context 是否被取消。如果已取消，立即中断推理并释放 GPU 显存。
> 3. **幂等性保障**：利用 RequestID 记录状态。如果用户断开后立即重连，我们会尝试从缓存中找回已生成的进度，而不是重新开始。”

### 场景二：分布式缓存一致性与 LLM 语义缓存
**面试官 (Expert)**：
> “你在 `learn.md` 中提到了缓存一致性要求。在 AI 问答中，如果规则或知识库（RAG）更新了，你如何确保所有分布式的缓存（包括内存缓存）能实时感知并失效？如果发生‘缓存击穿’，且这个请求是昂贵的 LLM 调用，你怎么处理？”

**候选人 (Senior Engineer)**：
> “针对缓存失效，我会采用 **‘订阅式失效’** 机制。利用 Pulsar/Redis 的广播频道，当知识库更新时，发布一个失效事件，各业务节点监听并清除本地内存（L1）缓存。
> 针对 LLM 的缓存击穿：
> 1. **分布式 Single-flight**：在 Redis 层实现一个带超时的占位锁。当第一个请求发现 Cache Miss 时，在 Redis 设置 `IN_PROGRESS` 标志。后续相同请求发现该标志后，进入退避重试或监听等待状态，直到第一个请求写回结果。
> 2. **语义化缓存**：针对 AI 场景，完全一致的 Key 命中率低。我们会引入 Vector Embedding，对用户提问做相似度计算（如余弦相似度 > 0.95），如果命中语义缓存则直接返回，大幅降低成本。”
