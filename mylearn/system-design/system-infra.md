# 异构超算基础设施面试指南

本文档针对 JD 四（异构超算基础设施），侧重于大模型算力底座的构建。这属于系统工程的最前沿，需要展现对硬件架构（GPU/NPU）、互联协议和大规模集群管理的深度理解。

## 1. 核心摘要与深度细节补充

### 1.1 异构资源池化 (GPU/NPU Pooling)
*   **细节补充**：如何突破单机 8 卡的限制？**MIG** (Multi-Instance GPU) 与 **vGPU** 的区别。如何实现显存的非侵入式监控和超卖（Overcommit）？
*   **虚拟化开销**：在异构计算中，任何额外的抽象层（如虚拟化）都会显著降低吞吐。如何利用 **GPUDirect RDMA** 绕过 CPU 内存直接进行 GPU 间通信？

### 1.2 通信优化与互联
*   **细节补充**：**NCCL** (NVIDIA Collective Communications Library) 的核心算子（AllReduce, AllGather）原理。**RDMA** (RoCE v2) 在大规模集群中的拥塞控制（PFC, ECN）。
*   **拓扑感知**：调度器如何感知胖树（Fat-Tree）拓扑，确保需要高频通信的 GPU 处于同一交换机下。

### 1.3 故障容错 (Fault Tolerance)
*   **细节补充**：大模型训练耗时长，单点故障风险高。**Checkpointing** 的 IO 瓶颈如何解决？（分布式快照、内存镜像）。

## 2. 面试对垒 (Expert vs. Senior)

### 场景一：GPU 资源利用率与池化调度
**面试官 (Expert)**：
> “我们的集群有数千张异构算力卡。目前的痛点是：部分任务只需要少量显存，却占用了一整张卡，导致资源浪费；而核心训练任务又常因为卡间通信带宽不足而变慢。你如何设计一套‘资源编排与池化系统’？”

**候选人 (Senior Engineer)**：
> “我会从**‘精细化切分’**与**‘拓扑亲和性调度’**两个维度入手：
> 1. **算力切分 (GPU Partitioning)**：
>    *   针对推理和测试任务，利用 NVIDIA 的 **MIG** 或自研的 API 拦截（Hook）技术实现显存和算力的虚拟化，将物理卡切分为多个 vGPU，满足多租户需求。
>    *   针对训练任务，强制独占模式，防止干扰。
> 2. **拓扑感知调度 (Topology-Aware Scheduling)**：
>    *   在调度器中构建全量的硬件拓扑树。利用 Kubernetes 的自定义调度器插件，确保执行同步并行训练（Data Parallelism）的 Pod 被优先分配到同一机架内，并尽量利用 **NVLink** 进行卡间互联。
> 3. **资源池化**：设计一个‘算力抽象层’。业务层不再申请‘张’卡，而是申请‘TFLOPS’和‘GB 显存’。底层通过驱动层动态绑定物理资源，并在空闲时自动回收。”

### 场景二：大规模集群通信性能调优
**面试官 (Expert)**：
> “在千卡规模的并行推理/训练中，网络延迟（Tail Latency）往往成为瓶颈。如果 NCCL 通信出现毛刺，你会如何利用 Profiling 工具定位问题？是驱动问题、硬件故障还是网络拥塞？”

**候选人 (Senior Engineer)**：
> “这需要全链路的可观测性手段：
> 1. **多维度 Profiling**：
>    *   首先使用 **NSIGHT Systems** 或 **PyTorch Profiler** 查看 GPU 时间线，确认是否为‘计算/通信重叠（Overlap）’做得不够好导致通信等待。
>    *   使用 `nccl-tests` 独立测试集群带宽，排除驱动版本不一致或 NCCL 策略选型（如 Ring vs. Tree）的问题。
> 2. **网络侧诊断**：
>    *   查看 RDMA 网卡的硬件计数器（Retransmit, PFC Pause Frames）。如果有大量 Pause 帧，说明网络存在拥塞。
>    *   我会利用 eBPF 监控内核态的网络调用栈，排查是否因为 CPU 调度导致的中断延迟影响了 RDMA 性能。
> 3. **故障预判**：
>    *   通过采集显卡 XID 错误日志和温度、功耗数据，建立异常检测模型。在硬件彻底失效前，利用迁移机制将任务转移，避免训练因单个显卡硬件故障（如 ECC 错误）而中断复位。”

**追问 (Expert)**：
> “当遇到跨硬件平台（如混合使用 NVIDIA 和华为 NPU）时，如何实现统一的算力调度和通信优化？”

**回答 (Senior Engineer)**：
> “这是‘异构计算’最难的一环。
> 1. **抽象框架**：采用 **OneCCL** 或类似的跨厂商通信库，或者在自研平台上开发兼容 NCCL/HCCL 的抽象层，屏蔽底层互联协议的差异。
> 2. **统一镜像**：利用多架构容器镜像，并配合不同的设备插件（Device Plugin）在启动时注入对应的运行时库。
> 3. **算力归一化**：建立一套 Benchmark 体系，将不同平台的算力折算为统一的‘单位算力（Credit）’，作为调度器的统一分配权重。”

## 3. 针对 JD 的追问方向（自测）
1. **能效比提升**：在超算环境下，如何通过动态调整 GPU 频率（DVFS）在保证性能的前提下降低能耗？
2. **零拷贝技术**：在大规模数据加载阶段，如何结合 **GPUDirect Storage (GDS)** 绕过内存直接从存储读入 GPU？
3. **分布式存储**：在 Checkpoint 场景下，传统的 POSIX 文件系统在高并发写入时如何扩容？（考虑 Lustre 或自研并行文件系统）
