# AI 异步工作流与高性能消息底座指南

本文档深入探讨史斌 CV 中的“AI 审核 Agent”架构，重点解决 LLM 的慢速响应与高并发业务流量之间的结构性矛盾。

## 1. 核心技术深度拆解

### 1.1 异步分层漏斗架构
*   **挑战**：LLM 极其昂贵且慢。如果 100% 的流量都过模型，成本会崩盘且延迟不可控。
*   **细节**：
    *   **一阶段（关键词/正则）**：由高性能 Go 引擎完成（P99 < 1ms），过滤掉 80% 的确定性黑白数据。
    *   **二阶段（轻量级模型）**：如 FastText 或自研小型 CNN，过滤 15% 的疑似数据。
    *   **三阶段（LLM Agent）**：仅对剩下的 5% 极具争议的语义进行深度研判。
*   **调度机制**：利用 Pulsar 的 **Delayed Message** 和 **Shared Subscription** 实现任务的动态分发。

### 1.2 处理 LLM 的“非确定性”
*   **细节**：LLM 有时会幻觉、超时或返回格式错误。
    *   **状态机驱动**：使用 FSM（有限状态机）维护工单状态（Init -> LLM_Processing -> Manual_Review -> Finished）。
    *   **死信队列 (DLQ) 升级**：对于格式报错，自动重试；对于连续 3 次失败，自动降级为“人审”并标记为 Badcase 记录。

## 2. 专家级面试对垒

### 场景一：Pulsar 削峰填谷与背压（Backpressure）
**面试官 (Expert)**：
> “突发热点事件导致审核流量瞬间翻了 10 倍。虽然 Pulsar 能积压消息，但如果下游的 LLM 调用（通过 API 或自建集群）已经满负荷，你会如何设计背压机制？如何防止积压的消息在恢复后由于‘过期’而失去审核意义？”

**候选人 (Senior Engineer)**：
> “我会设计一套**‘基于消息 TTL 与算力反馈的自适应流控’**系统：
> 1. **算力感知推送**：Worker 节点不再是被动接收推送，而是根据自身的并发处理能力（Active Goroutines）和 LLM API 的 Quota 限制，从 Pulsar 主动 `Pull` 消息。
> 2. **消息时效性控制**：
>    *   在消息属性中注入 `ExpiryTime`。
>    *   Worker 拉取后，首先检查当前时间是否已超过过期时间。如果已过期（说明是数小时前的积压），则根据业务策略直接通过或走最简单的静态规则（降级）。
> 3. **动态分区缩容**：利用 Pulsar 的分区机制。如果积压过大，动态增加 Consumer 数量；如果 LLM 侧链路挂掉，立即触发网关层的**熔断限流**，在最前端把压力顶回去（丢弃低价值流量）。”

### 场景二：保障分布式链路下的数据强一致性
**面试官 (Expert)**：
> “你的流程经历了‘机审-人审-回流’。如果消息在 Pulsar 中被消费了，但在写入 TiDB 状态时失败了，或者人审平台完成了操作但回调消息丢失了，你会看到‘审核中’的状态一直悬挂。你如何实现零资损、强一致的流转？”

**候选人 (Senior Engineer)**：
> “我通过 **‘事务消息 + 幂等性位移’** 来解决：
> 1. **Pulsar 事务消息**：利用 Pulsar 2.8+ 的 Transaction 功能，确保‘消息消费’与‘DB 状态更新’在一个原子操作内。
> 2. **业务幂等标识**：每个工单都有唯一的 `TraceID`。在 DB 侧使用 `UPDATE status = 'NEW' WHERE status = 'OLD' AND trace_id = 'xxx'` 的乐观锁模式。
> 3. **状态主动回补（Self-Healing）**：
>    *   我会启动一个 CronJob 扫描 TiDB 中处于‘中间状态’超过 10 分钟的任务。
>    *   主动向人审平台或 LLM 记录查询结果，如果对方已完成，则本地补推状态；如果对方无记录，则重发 Pulsar 消息再次触发任务。这保证了**最终一致性**。”
