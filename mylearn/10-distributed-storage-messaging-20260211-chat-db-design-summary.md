## 3. 核心技术决策

### 3.1 分片策略选择

| 表名 | 分片键 | 理由 |
|------|--------|------|
| messages | conv_id | 查询模式匹配，单分片查询 |
| user_conversations | user_id | 用户会话列表查询，高效分页 |
| conversations | conv_id | 会话基本信息，与messages关联 |
| conversation_members | user_id | 用户成员关系，避免跨分片 |
| conversation_context | conv_id | AI上下文与会话绑定 |
| tool_calls | conv_id | 工具调用与会话绑定 |
| token_usage | user_id | 用户Token统计，按用户查询 |

### 3.2 时间分片策略选择

**推荐方案**: 不使用时间分片作为主分片策略

**原因**:
1. 查询模式不匹配：Chat应用按会话查询，不是按时间查询
2. 性能问题：需要扫描所有时间分区
3. 成本问题：计算成本最高

**替代方案**: 使用归档表实现冷热分离
- 主表：messages按conv_id分片（热数据）
- 归档表：messages_archive按时间分片（冷数据）

### 3.3 成本优化策略

**推荐方案**: Conv_id分片 + 冷热分离

**成本对比**:
- 时间分片：$8632/月（存储成本低，计算成本高）
- Conv_id分片：$9900/月（存储成本高，计算成本低）
- 混合方案：$9840/月（平衡方案）

**优化策略**:
1. 压缩优化：使用MySQL压缩，节省67%存储成本
2. 分层存储：NVMe SSD → SATA SSD → HDD
3. 数据生命周期管理：自动迁移旧数据

## 5. 核心知识点总结

### 5.1 MySQL核心机制

**MVCC机制**:
- 通过undo log实现多版本并发控制
- Read View决定数据可见性
- 适合读多写少场景

**锁机制**:
- S锁（共享锁）：读锁
- X锁（排他锁）：写锁
- Gap锁：防止幻读
- 意向锁：提高锁粒度

**索引优化**:
- B+树结构，适合范围查询
- 覆盖索引减少回表
- 索引下推优化查询
- 前缀索引节省空间

### 5.2 Chat应用数据库设计

**核心原则**:
1. **查询模式优先**：选择与查询模式匹配的分片策略
2. **性能优先**：用户体验比存储成本更重要
3. **冷热分离**：通过归档表实现，而不是时间分片
4. **成本可控**：通过压缩、分层存储、数据生命周期管理优化

**推荐架构**:
```
主表（热数据）：
- messages: 按conv_id分片
- user_conversations: 按user_id分片
- conversations: 按conv_id分片

归档表（冷数据）：
- messages_archive: 按时间分片
```

### 5.3 分片策略对比

| 维度 | user_id分片 | conv_id分片 | 时间分片 |
|------|-------------|-------------|----------|
| 查询性能 | 跨分片扫描，O(n) | 单分片查询，O(1) | 跨分区扫描，O(n) |

| 分页性能 | 跨分片排序，性能差 | 单分片排序，性能好 | 跨分区排序，性能差 |
| 数据一致性 | 需分布式事务 | 单数据库事务 | 单数据库事务 |
| 存储成本 | 高（群聊冗余） | 低（无冗余） | 中等（冷热分离） |
| 适用场景 | 个人消息查询 | 会话内消息查询 | 时间范围查询 |

## 7. 关键决策点回顾

### 7.1 为什么messages表按conv_id分片？

**原因**:
1. **查询模式匹配**：Chat应用主要按会话查询消息
2. **性能优势**：单分片查询 vs 跨分片扫描
3. **写入效率**：消息发送只需插入一次
4. **数据一致性**：单数据库事务，避免分布式事务

**对比user_id分片**:
- user_id分片会导致同一个会话的消息分散在多个分片
- 查询会话消息需要扫描所有分片，性能极差
- 群聊场景需要多次插入，数据冗余

**对比时间分片**:
- 时间分片会导致同一个会话的消息分布在多个时间分区
- 查询会话消息需要扫描所有时间分区，性能极差
- 适合归档数据，不适合实时查询

### 7.2 为什么user_conversations表按user_id分片？

**原因**:
1. **查询模式匹配**：用户查看自己的会话列表
2. **高效分页**：单分片内排序分页
3. **性能优化**：避免跨分片扫描

**设计要点**:
- 包含last_msg_time、unread_count等字段
- 支持按时间排序、按未读数排序
- 支持分页查询（LIMIT/OFFSET或游标分页）

### 7.3 为什么冷热分离不推荐时间分片？

**原因**:
1. **查询模式不匹配**：Chat应用按会话查询，不是按时间查询
2. **性能问题**：需要扫描所有时间分区
3. **成本问题**：计算成本最高

**推荐方案**:
- 主表：messages按conv_id分片（热数据）
- 归档表：messages_archive按时间分片（冷数据）
- 通过定期迁移实现冷热分离

## 8. 成本优化策略

### 8.1 压缩优化

```sql
CREATE TABLE messages (
    ...
) ENGINE=InnoDB
ROW_FORMAT=COMPRESSED
KEY_BLOCK_SIZE=8;
```

**效果**: 1TB → 333GB，存储成本$33/月
**节省**: 67%存储成本

### 8.2 分层存储

```
NVMe SSD（热数据，7天内）：$0.15/GB/月
SATA SSD（温数据，7-90天）：$0.10/GB/月
HDD（冷数据，90天以上）：$0.02/GB/月
```

**效果**: 比全SSD节省60-80%

### 8.3 数据生命周期管理

```sql
-- 自动迁移策略
-- 7天前：NVMe SSD → SATA SSD
-- 90天前：SATA SSD → HDD
-- 1年前：HDD → 对象存储（S3）
```

## 9. 总结与展望

### 9.1 核心结论

1. **messages表应该按conv_id分片，而不是user_id或时间分片**
   - 查询模式匹配：Chat应用按会话查询
   - 性能优势：单分片查询 vs 跨分片扫描
   - 写入效率：消息发送只需插入一次
   - 数据一致性：单数据库事务

2. **时间分片的冷热分离是优势，但查询模式不匹配是致命缺陷**
   - 存储成本最低，但计算成本最高
   - 查询性能差20倍，用户体验差
   - 适合归档数据，不适合实时查询

3. **混合方案（Conv_id分片 + 冷热分离）是最佳平衡**
   - 查询性能最好：会话内查询10ms-
   - 存储成本可控：比Conv_id分片低49%
   - 冷热分离：主表热数据，归档表冷数据

### 9.2 关键原则

1. **查询模式优先**：选择与查询模式匹配的分片策略
2. **性能优先**：用户体验比存储成本更重要
3. **冷热分离**：通过归档表实现，而不是时间分片
4. **成本可控**：通过压缩、分层存储、数据生命周期管理优化

4. **自动化运维**：如何自动化分区管理和数据归档
